# SnapInsight API Specification

## Document Status

- Status: Approved
- Related Documents:
  - `docs/prd/PRD-snapinsight.md`
  - `docs/rfcs/RFC-002-local-communication-and-security.md`
  - `docs/rfcs/RFC-003-python-service-and-ollama-integration.md`
  - `docs/design/extension-and-local-service-design.md`

## 1. Purpose

This document defines the v1 interface contracts between:

- the Chrome extension service worker and the local Python service
- the content script and the extension service worker

The goal is to make request and response formats explicit enough for implementation without reopening the design decisions already accepted in the RFCs.

## 2. Scope

This spec covers:

- local HTTP API on `127.0.0.1:11435`
- request and response schemas for health, model list, and explanation streaming
- streaming event format
- machine-readable error codes
- internal extension message contracts

This spec does not define:

- prompt template internals
- Ollama upstream API details
- exact UI text shown to end users

## 3. Transport Rules

### 3.1 Local HTTP Boundary

- Base URL: `http://127.0.0.1:11435`
- Allowed caller: Chrome extension `service worker`
- Content script must not call the local API directly
- Local service must only bind to `127.0.0.1`
- Local service must validate allowed `chrome-extension://<extension-id>` origins

### 3.2 Content Types

- Standard JSON endpoints use `application/json`
- Streaming explanation endpoint uses `application/x-ndjson`

### 3.3 Versioning

- v1 endpoints use `/v1/` where applicable
- `GET /health` remains unversioned for operational convenience

### 3.4 Failure Boundary Rules

- HTTP status codes represent failures detected before a streaming response is successfully established.
- Once a streaming response has begun with `200 OK`, later failures must be expressed with a terminal stream `error` event.
- Callers must not expect the service to switch to a new HTTP status after a `200` stream has already started.

## 4. Common Data Types

### 4.1 Error Object

```json
{
  "code": "request_failed",
  "message": "Explanation generation failed.",
  "retryable": true
}
```

Fields:

- `code`: stable machine-readable error code
- `message`: implementation-facing or extension-facing message, not raw stack trace
- `retryable`: whether the caller may surface a retry action

### 4.2 Model Summary

```json
{
  "id": "llama3.1:8b",
  "label": "llama3.1:8b",
  "provider": "ollama",
  "available": true
}
```

Fields:

- `id`: canonical model identifier used in requests
- `label`: UI-friendly display label
- `provider`: fixed to `ollama` in v1
- `available`: whether the model is currently available for selection

### 4.3 Explanation Mode

Allowed values:

- `short`
- `detailed`

### 4.4 Sender Context

```json
{
  "tabId": 123,
  "frameId": 0,
  "pageInstanceId": "doc-7f6d6b2d"
}
```

Fields:

- `tabId`: originating browser tab identity
- `frameId`: originating frame identity when available
- `pageInstanceId`: per-document identifier generated by the content script for one loaded page instance

Rules:

- sender context is required for extension-internal stream routing
- `requestId` is not sufficient on its own for routing or cancellation
- a new `pageInstanceId` must be created on reload or same-tab navigation so stale streams cannot bind to a new document instance

## 5. HTTP API

### 5.1 `GET /health`

Purpose:

- determine whether the local SnapInsight service is reachable
- help the extension distinguish service availability from model availability

Success response:

```json
{
  "status": "ok",
  "service": "snapinsight-local-api",
  "version": "v1",
  "ollamaReachable": true
}
```

Response fields:

- `status`: must be `ok` when the service is healthy enough to answer requests
- `service`: fixed service identifier
- `version`: current contract version
- `ollamaReachable`: whether the service can currently reach Ollama

Behavior rules:

- `GET /health` is the source of truth for SnapInsight local service reachability.
- `ollamaReachable` is dependency context only; it does not replace `GET /v1/models` as the source of truth for model availability.
- the extension must verify `service == "snapinsight-local-api"` before trusting the fixed localhost port as the SnapInsight service
- if the port responds but the `service` field is missing or mismatched, the extension must treat the result as a local service conflict and fail closed

Status codes:

- `200 OK`: service is reachable

Failure handling:

- if the HTTP request cannot be established at all, the extension service worker must normalize the result to `service_unavailable`

### 5.2 `GET /v1/models`

Purpose:

- return the currently available models from Ollama in a stable SnapInsight format

Success response with models:

```json
{
  "state": "ready",
  "models": [
    {
      "id": "llama3.1:8b",
      "label": "llama3.1:8b",
      "provider": "ollama",
      "available": true
    }
  ]
}
```

Success response with no models:

```json
{
  "state": "no_models_available",
  "models": []
}
```

Response fields:

- `state`: `ready` or `no_models_available`
- `models`: list of available models

Status codes:

- `200 OK`: request handled successfully, including the case where no model is available
- `503 Service Unavailable`: Ollama could not be queried even though the local service is running
- `500 Internal Server Error`: unexpected local service failure

Behavior rules:

- raw Ollama response payloads must not be forwarded directly
- if Ollama is reachable but has no models, return `200` with `state: "no_models_available"`
- if Ollama itself is unavailable, the service must return `503` with:

```json
{
  "error": {
    "code": "request_failed",
    "message": "Model list could not be loaded.",
    "retryable": true
  }
}
```

The extension may map this result to a settings or in-card error depending on current context.

Interpretation rules:

- `GET /health` returning `200` means the SnapInsight local service is alive.
- `GET /v1/models` returning `200` with `state: "no_models_available"` means Ollama is reachable but has no selectable models.
- `GET /v1/models` returning `503` with `error.code: "request_failed"` means the SnapInsight local service is alive but cannot currently query Ollama.

### 5.3 `POST /v1/explanations/stream`

Purpose:

- request a streamed explanation for the selected text and chosen model

Request body:

```json
{
  "requestId": "9e86d5ac-35a7-4a59-8d57-8db1f71db9f6",
  "text": "Transformer",
  "model": "llama3.1:8b",
  "mode": "short"
}
```

Request fields:

- `requestId`: caller-generated unique ID for tracing and cancellation alignment
- `text`: selected text to explain
- `model`: chosen model identifier
- `mode`: `short` or `detailed`

Validation rules:

- `requestId` must be a non-empty string
- `text` must be a non-empty trimmed string
- the extension product flow must only send selections that satisfy the PRD-level `1-20` product limit
- the local service may additionally reject any payload whose `text` exceeds `200` Unicode characters as a defensive API ceiling
- `model` must be a non-empty string
- `mode` must be one of the allowed explanation modes

Status codes:

- `200 OK`: stream established successfully
- `400 Bad Request`: invalid request payload detected before stream establishment
- `403 Forbidden`: origin not allowed
- `409 Conflict`: selected model unavailable before stream establishment
- `503 Service Unavailable`: required local dependency unavailable before stream establishment
- `500 Internal Server Error`: unexpected failure before stream establishment

Response body:

- response is a newline-delimited JSON stream with content type `application/x-ndjson`
- each line is one JSON object
- event order must be:
  1. exactly one `start`
  2. zero or more `chunk`
  3. exactly one terminal event: `complete` or `error`

Failure rules:

- if a failure is known before the server starts the stream, return the corresponding HTTP status with a standard JSON `error` body and do not begin the NDJSON stream
- if a failure happens after the stream has started, keep the HTTP status at `200` and terminate the stream with an `error` event

Product counting rules for the `1-20` limit:

- trim leading and trailing whitespace before validation
- strip leading and trailing punctuation from the full selection
- split the remaining text by whitespace into segments
- for a segment containing only Latin letters, digits, connector punctuation such as `-` and `'`, count the segment as `1`
- for a segment containing CJK characters, count each visible CJK character as `1`
- for a segment containing both CJK characters and a contiguous Latin/digit run without spaces, count the Latin/digit run as `1` and each CJK character as `1`
- the mixed counting rule is a product-side UI rule and does not change the defensive API ceiling

Examples:

- `GPT-4` counts as `1`
- `RAG-based agent` counts as `2`
- `AI大模型` counts as `5`

## 6. Streaming Event Schema

### 6.1 `start` Event

```json
{
  "event": "start",
  "requestId": "9e86d5ac-35a7-4a59-8d57-8db1f71db9f6",
  "mode": "short",
  "model": "llama3.1:8b"
}
```

### 6.2 `chunk` Event

```json
{
  "event": "chunk",
  "requestId": "9e86d5ac-35a7-4a59-8d57-8db1f71db9f6",
  "delta": "一种用于处理序列数据的神经网络架构"
}
```

Rules:

- `delta` contains the next text fragment to append
- chunks must arrive in display order
- the client should append `delta` as received without waiting for stream completion
- internal worker-to-content-script delivery may coalesce multiple small upstream chunks into fewer chunk messages, as long as ordered progressive rendering remains possible and terminal-event semantics are unchanged

### 6.3 `complete` Event

```json
{
  "event": "complete",
  "requestId": "9e86d5ac-35a7-4a59-8d57-8db1f71db9f6"
}
```

Rules:

- `complete` is the only successful terminal event
- no additional events may follow `complete`

### 6.4 `error` Event

```json
{
  "event": "error",
  "requestId": "9e86d5ac-35a7-4a59-8d57-8db1f71db9f6",
  "error": {
    "code": "selected_model_unavailable",
    "message": "The selected model is no longer available.",
    "retryable": false
  }
}
```

Rules:

- `error` is the only failure terminal event
- no additional events may follow `error`
- if some text was already streamed before the error, the client may keep already rendered text and mark only the affected area as failed
- `error.code` must use one of the public error codes defined in Section 7

## 7. Error Codes

The following machine-readable error codes are part of the v1 contract:

| Code | Meaning | Typical caller behavior |
|------|---------|-------------------------|
| `service_unavailable` | The local HTTP service could not be reached | show local-service-not-running state |
| `local_service_conflict` | The fixed localhost port responded, but it was not the SnapInsight local service | show local setup conflict guidance and fail closed |
| `invalid_request` | Request payload failed validation | treat as developer or implementation error |
| `no_models_available` | Ollama is reachable but no selectable model exists | block explanation requests and ask user to configure a model |
| `selected_model_unavailable` | The user's stored model no longer exists | require model re-selection |
| `request_failed` | Explanation or model-list request failed unexpectedly | show retryable error where appropriate |
| `request_cancelled` | Request ended due to user-driven cancellation or disconnect | usually suppress visible error |

Rules:

- extension transport failures must be normalized to `service_unavailable`
- Python stack traces and raw upstream errors must not be exposed as `message`
- if more internal error categories are needed, they must still map into one of the v1 public codes above
- `request_failed` is the public error code used when Ollama is unavailable but the SnapInsight local service itself is reachable
- `local_service_conflict` is the public error code used when the fixed localhost port responds with a non-SnapInsight service identity
- `request_cancelled` should only be emitted when cancellation is explicitly surfaced across the extension boundary; silent user-driven cancellation does not require a visible terminal message

## 8. Extension Internal Message Contracts

The content script and service worker communicate through structured internal messages. The exact transport API may be `chrome.runtime.sendMessage` plus a long-lived stream bridge, but the message shapes below are the contract boundary.

### 8.1 Health Check

Request:

```json
{
  "type": "health.check"
}
```

Response:

```json
{
  "ok": true,
  "data": {
    "status": "ok",
    "service": "snapinsight-local-api",
    "ollamaReachable": true
  }
}
```

If the fixed port responds with the wrong service:

```json
{
  "ok": false,
  "error": {
    "code": "local_service_conflict",
    "message": "The expected localhost port is occupied by a different service.",
    "retryable": false
  }
}
```

### 8.2 Model List Fetch

Request:

```json
{
  "type": "models.list"
}
```

Response:

```json
{
  "ok": true,
  "data": {
    "state": "ready",
    "models": [
      {
        "id": "llama3.1:8b",
        "label": "llama3.1:8b",
        "provider": "ollama",
        "available": true
      }
    ]
  }
}
```

If no model is available:

```json
{
  "ok": true,
  "data": {
    "state": "no_models_available",
    "models": []
  }
}
```

If Ollama cannot be queried:

```json
{
  "ok": false,
  "error": {
    "code": "request_failed",
    "message": "Model list could not be loaded.",
    "retryable": true
  }
}
```

If the fixed port responds with the wrong service:

```json
{
  "ok": false,
  "error": {
    "code": "local_service_conflict",
    "message": "The expected localhost port is occupied by a different service.",
    "retryable": false
  }
}
```

### 8.3 Read Selected Model

Request:

```json
{
  "type": "settings.getSelectedModel"
}
```

Response:

```json
{
  "ok": true,
  "data": {
    "selectedModel": "llama3.1:8b",
    "lastKnownModels": [
      {
        "id": "llama3.1:8b",
        "label": "llama3.1:8b",
        "provider": "ollama",
        "available": true
      }
    ],
    "lastModelRefreshAt": "2026-04-10T10:00:00.000Z"
  }
}
```

If no model is selected yet:

```json
{
  "ok": true,
  "data": {
    "selectedModel": null,
    "lastKnownModels": [],
    "lastModelRefreshAt": null
  }
}
```

Rules:

- `settings.getSelectedModel` is a read-only convenience contract for settings rendering and blocked-setup hints
- the response may also include `lastKnownModels` and `lastModelRefreshAt` as worker-backed diagnostic convenience fields for settings rendering when live model loading later fails
- these diagnostic fields must reflect the same persisted settings snapshot owned by the worker and must not become an authoritative substitute for live model validation
- the in-page hover-triggered explanation flow does not need `settings.getSelectedModel` to succeed before it is allowed to call `explanations.start`
- authoritative startup validation for an explanation attempt may be completed inside `explanations.start`

### 8.4 Update Selected Model

Request:

```json
{
  "type": "settings.setSelectedModel",
  "payload": {
    "selectedModel": "llama3.1:8b"
  }
}
```

Validation rules:

- `selectedModel` must match one of the currently available models returned by `models.list`
- the service worker should not persist a model value that is already known to be unavailable

Response:

```json
{
  "ok": true
}
```

If the selected model is no longer available:

```json
{
  "ok": false,
  "error": {
    "code": "selected_model_unavailable",
    "message": "The selected model is no longer available.",
    "retryable": false
  }
}
```

If the local service cannot be reached while validating the selection:

```json
{
  "ok": false,
  "error": {
    "code": "service_unavailable",
    "message": "The local service could not be reached.",
    "retryable": true
  }
}
```

If the fixed port responds with the wrong service:

```json
{
  "ok": false,
  "error": {
    "code": "local_service_conflict",
    "message": "The expected localhost port is occupied by a different service.",
    "retryable": false
  }
}
```

If the model catalog cannot be loaded for another retryable reason:

```json
{
  "ok": false,
  "error": {
    "code": "request_failed",
    "message": "The selected model could not be validated.",
    "retryable": true
  }
}
```

### 8.5 Start Explanation Stream

Request:

```json
{
  "type": "explanations.start",
  "payload": {
    "requestId": "9e86d5ac-35a7-4a59-8d57-8db1f71db9f6",
    "senderContext": {
      "tabId": 123,
      "frameId": 0,
      "pageInstanceId": "doc-7f6d6b2d"
    },
    "text": "Transformer",
    "mode": "short"
  }
}
```

Alternative request with explicit model override:

```json
{
  "type": "explanations.start",
  "payload": {
    "requestId": "9e86d5ac-35a7-4a59-8d57-8db1f71db9f6",
    "senderContext": {
      "tabId": 123,
      "frameId": 0,
      "pageInstanceId": "doc-7f6d6b2d"
    },
    "text": "Transformer",
    "model": "llama3.1:8b",
    "mode": "short"
  }
}
```

Response:

```json
{
  "ok": true,
  "data": {
    "requestId": "9e86d5ac-35a7-4a59-8d57-8db1f71db9f6"
  }
}
```

Rules:

- a successful `explanations.start` response only confirms that the extension bridge and startup request were accepted
- the actual stream lifecycle begins when the forwarded internal `start` event is emitted
- `payload.model` is optional in the extension-internal contract
- if `payload.model` is omitted, the service worker must resolve the effective model from current persisted extension settings before stream establishment rather than trying to infer card-local UI state that it does not own
- if `payload.model` is present, the service worker must validate it as an explicit model override before stream establishment
- once an open card interaction already has an established effective model, same-card detail requests and same-card retries should pass that model explicitly in `payload.model` so the card-scoped model snapshot is reused instead of being silently replaced by later global settings changes
- the worker-side omission rule does not relax the content-script obligation above; same-card model stability depends on the content script passing `payload.model` once the card already has an established `activeModel`

If setup fails because the local service cannot be reached before the stream starts:

```json
{
  "ok": false,
  "error": {
    "code": "service_unavailable",
    "message": "The local service could not be reached.",
    "retryable": true
  }
}
```

If setup fails for another retryable startup or dependency reason before the stream starts:

```json
{
  "ok": false,
  "error": {
    "code": "request_failed",
    "message": "Explanation stream could not be started.",
    "retryable": true
  }
}
```

If setup fails because no usable selected model currently exists:

```json
{
  "ok": false,
  "error": {
    "code": "selected_model_unavailable",
    "message": "A valid model must be selected before explanation can start.",
    "retryable": false
  }
}
```

If setup fails because the fixed port responds with the wrong service:

```json
{
  "ok": false,
  "error": {
    "code": "local_service_conflict",
    "message": "The expected localhost port is occupied by a different service.",
    "retryable": false
  }
}
```

Setup-time failure mapping:

| Detected condition | Internal error code |
|------|---------|
| Local service transport failure | `service_unavailable` |
| Origin rejected before stream establishment | `request_failed` |
| Wrong-service identity on fixed port | `local_service_conflict` |
| No usable selected model exists, or selected model is unavailable or rejected at startup | `selected_model_unavailable` |
| Other startup or dependency failure | `request_failed` |

Streaming delivery rule:

- after accepting `explanations.start`, the service worker must forward stream events back to the content script using an event channel keyed by `requestId` plus `senderContext`
- forwarded events must preserve the API event contract from Section 6
- if the underlying HTTP request fails before stream establishment, the service worker must convert that HTTP failure into a non-streaming internal error result instead of pretending that a stream exists
- the bridge carrying stream events must remain active for the lifetime of the request
- if the MV3 bridge is lost unexpectedly, the service worker or content script must transition the affected request to a retryable failure state
- unexpected bridge loss after stream acceptance must be normalized to `request_failed` with `retryable: true`
- for in-page explanation attempts, `explanations.start` serves as the single authoritative startup path even when the UI already has cached or previously loaded selected-model state
- the extension should not require a separately successful `settings.getSelectedModel` call before issuing `explanations.start`

```json

  "type": "explanations.event",
  "payload": {

    "senderContext": {
      "tabId": 123,
      "frameId": 0,

    "event": {
      "event": "error",
      "requestId": "9e86d5ac-35a7-4a59-8d57-8db1f71db9f6",
      "error": {
        "code": "request_cancelled",
        "message": "The request was cancelled.",
        "retryable": false
      }
    }
  }
}
```

Rules:

- the internal envelope type must be `explanations.event`
- `payload.senderContext` must match the scoped interaction that started the request
- `payload.event` must preserve the Section 6 stream-event contract
- bridge loss must be surfaced through the same internal event envelope, normalized as retryable `request_failed`
- if explicit cancellation is surfaced, it must use the same internal envelope with terminal error code `request_cancelled`
- the forwarded `start` event must not be omitted when the stream is actually established

### 8.7 Cancel Explanation Stream

Request:

```json
{
  "type": "explanations.cancel",
  "payload": {
    "requestId": "9e86d5ac-35a7-4a59-8d57-8db1f71db9f6",
    "senderContext": {
      "tabId": 123,
      "frameId": 0,
      "pageInstanceId": "doc-7f6d6b2d"
    }
  }
}
```

Response:

```json
{
  "ok": true
}
```

Rules:

- cancellation is best-effort
- a cancelled request may end without any additional forwarded event
- the UI should treat user-driven cancellation as a silent state transition, not as a visible failure
- cancellation must be scoped to the same `requestId` plus `senderContext` pair used when the stream was started

## 9. Privacy and Logging Rules

- selected text must only travel from content script to service worker to local service
- selected text must not be persisted in extension storage
- selected text and generated output must not be written to normal operational logs
- debug logging, if temporarily enabled during development, must be opt-in and excluded from default behavior

## 10. Residual Security Risk

- In v1, the fixed-port service-identity check reduces accidental port conflicts and ordinary wrong-service cases, but it does not fully authenticate the local process behind the port.
- A malicious local process could still impersonate the documented SnapInsight service identity and receive selected text.
- This residual risk is accepted for v1 and should be revisited if the project later requires stronger local-process trust guarantees.
